# 博客图片预览架构说明

这篇文档专门记录博客正文图片、图片组和 lightbox 预览的实现方式，以及这次修复过程中确认过的几个关键坑点。后续如果再改这块，先看这里，再动代码。

## 目标

- 点击正文图片或 gallery 图片时，必须打开用户实际点中的那一张。
- 移动端支持滑动切换，不依赖左右按钮。
- 页面滚动和 lightbox 打开不能互相污染，不能因为重渲染造成图片重复请求。
- 定位逻辑必须是确定性的，不能“猜”一张图，更不能随机兜底。

## 组件分层

- `Markdown.tsx` 负责把 MDX 标记映射成具体组件。
- `MarkdownMedia.tsx` 负责图片、图库和媒体卡片的交互实现。
- `ArticlePage.tsx` 负责 article 级别的 lightbox 状态、定位和切换。
- `MarkdownLightbox.tsx` 负责 lightbox 的展示和切换动画。

## 核心数据模型

lightbox 相关的数据都走 `LightboxState` / `LightboxImage`。

- `images` 是当前 lightbox 可切换的图片列表。
- `currentIndex` 是当前展示项。
- `selectedId` 是最可靠的定位字段，必须唯一且可追踪。
- `selectedIndex` 只能作为严格校验后的显式索引，不能当成兜底。
- `selectedImage` 只适合作为补充信息，不应该承担主要定位职责。

## 定位原则

### 1. 优先使用 `selectedId`

图片真正被点击时，应该带上稳定 id。lightbox 打开时先按 id 定位。

原因很简单：

- React 组件重渲染后，数组顺序和实例顺序都可能变化。
- `src`、`alt`、`caption` 不是稳定唯一键。
- gallery 内多张图经常出现重复或相似资源，靠内容匹配很容易命中错误项。

### 2. `selectedIndex` 必须严格校验

如果传了 `selectedIndex`：

- 必须是整数。
- 必须落在 `[0, images.length)` 里。
- 不合法直接报错。
- 不允许自动修正为第一张、最后一张或随机一张。

这是为了避免“看起来能用，但实际上点错图”的隐性错误。

### 3. 不允许模糊匹配兜底

以下做法都不推荐：

- 仅按 `src` 匹配。
- 用 `alt` / `caption` 拼条件猜测。
- 在找不到目标时回退到第一张图。

这类逻辑会把 bug 隐藏起来，最终表现就是“随机”。

## 这次问题的根因

这次踩坑主要有两层。

### 1. 组件身份不稳定会触发重复请求

如果在渲染过程中动态创建组件工厂，React 可能把它当成新的组件类型，导致整块子树重挂载。结果就是：

- 同一批图片重新请求。
- 首次渲染时同一张图可能加载多次。
- 打开 lightbox 或滚动页面时，正文内容反复重建。

解决方式是：

- MDX 组件映射要稳定。
- `openLightbox`、heading 生成器等回调要保持引用稳定。
- 不要在渲染路径里反复创建新的匿名组件类型。

### 2. lightbox 打开逻辑曾经有兜底分支

当时的问题不是 React `key`，而是打开逻辑里存在“单图时扩展到整篇文章图片”的兜底行为，同时 `selectedIndex` 又优先于 `selectedId`。

这会导致：

- 点击某张图时，最终定位到文章中的第一张。
- 多图文章里，点击和打开不一致。
- 由于数据来源不同，看起来像“随机”。

现在的原则是：只认明确命中的目标，不做猜测。

## 正确的数据流

### 单图点击

1. 图片按钮在渲染时携带自己的稳定 id。
2. 点击时只把这张图作为目标传给 lightbox。
3. `ArticlePage` 用 `selectedId` 找回对应项。
4. 找不到就报错，不兜底。

### gallery 点击

1. 每个 gallery item 都有自己的稳定 id。
2. 点击某个 item 时，传入当前 item 的 id 和当前图片对象。
3. `ArticlePage` 仍然优先按 id 定位。
4. 允许在同一个 gallery 列表里左右切换，但前提是定位必须准确。

## 实现约束

- 图片按钮和 gallery 按钮都必须带 `data-blog-lightbox-id`。
- `LightboxImage.id` 不能空着，尤其是会参与定位的时候。
- `selectedId` 和 `data-blog-lightbox-id` 必须一致。
- 任何新增图片入口，都要沿用同一套定位策略。
- 不要把“能打开”当成完成标准，必须验证“打开的是哪一张”。

## 常见错误

- 在渲染函数里直接创建新的组件工厂。
- 把 `selectedIndex` 当成默认主定位方式。
- 点击事件只传 `src`，不传稳定 id。
- 找不到目标后自动回退到第一张。
- 为了兼容而牺牲确定性。

## 修改前检查清单

改这块之前，至少确认以下几点：

- 新增的图片入口有没有稳定 id。
- lightbox 打开逻辑是不是优先按 id。
- `selectedIndex` 有没有做严格校验。
- 有没有任何“找不到就兜底”的分支。
- 是否会因为渲染重建导致图片重复请求。

## 结论

这块的核心不是“怎么显示图片”，而是“怎么稳定地识别用户点中的那一张图片”。

只要定位链路不稳定，后面所有按钮、动画、滑动体验都会跟着变得不可预测。以后改这里，优先保证数据确定性，再谈交互细节。
