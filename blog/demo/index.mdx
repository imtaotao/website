这篇文章专门用来放博客正文支持的各种 case。图片统一放在文章自己的 `image/` 文件夹里，正文直接写相对路径就行。

这是一段专门用于排版测试的中文说明。正文里如果同时出现中文、英文和代码片段，整体节奏和行高都应该保持自然，不要一眼看出是不同系统拼起来的。

## 1. 基础正文

普通段落、`inline code`、[站外链接](https://example.com) 都应该自然工作。**加粗**字体应该正常显示。

### 1.1 颜色示例

推荐用更短的写法 <Color c="green">`这是一段绿色文字`</Color>，正文里直接用预设颜色名就行。

目前内置了 8 个预设颜色：

- <Color c="gray">gray</Color>

- 一段简短说明
- 一组并列要点

## 3. 列表和代码块

```tsx {4-8,14,20-24} ln
import { Badge, Button, Stack } from 'willa';

type ArticleCardProps = {
  title: string;
  summary: string;
  tags: string[];
  href: string;
};

const ArticleCard = ({ title, summary, tags, href }: ArticleCardProps) => {
  const primaryTag = tags.at(0) ?? 'note';

  return (
    <article className="blog-demo-card" data-tag={primaryTag}>
      <Stack gap={12}>
        <Badge>{primaryTag}</Badge>
        <h3>{title}</h3>
        <p>{summary}</p>

        <div className="blog-demo-card__actions">
          <Button asChild size="sm">
            <a href={href}>阅读全文</a>
          </Button>
        </div>
      </Stack>
    </article>
  );
};
```

## 5. 数学表达式

行内公式可以直接写在正文里，比如欧拉恒等式 <MathExpression value="e^{i\pi} + 1 = 0" />，或者二次方程求根公式 <MathExpression value="x = \frac{-b \pm \sqrt{b^2 - 4ac}}{2a}" />。

块级公式适合推导和单独展示：

<MathExpression
  display="block"
  value="\int_{-\infty}^{\infty} e^{-x^2} \, dx = \sqrt{\pi}"
/>

多行公式也可以正常排版：

<MathExpression
  display="block"
  value="\begin{aligned}
f(x) &= ax^2 + bx + c \\
f'(x) &= 2ax + b
\end{aligned}"
/>

## 11. 阅读要点

如果一篇文章里有“讨论总结”“AI 提炼”“阅读要点”这类内容，优先用普通标题和列表承载。

### 第一点

- 摘要更适合放已经整理过的信息，而不是原始对话。
- 它会比普通段落更方便在长文里快速扫读。

### 第二点

- 如果文章后面还要继续贴聊天记录、访谈内容或者长段引用，可以先把结论放在前面。
- 这样读者先看到结构化信息，再决定要不要往下读原始材料。

## 13. 普通列表

列表适合写轻量说明，也可以保留行内 MDX 能力。

- 普通站外链接优先用 Markdown 链接，比如 [imtaotao.me](https://imtaotao.me)。
- 如果正文里要顺手提到 GitHub 用户或仓库，可以直接写 <GitHubMention username="imtaotao" /> 和 <GitHubRepo repo="imtaotao/website" />。
- 每一项里都可以直接放 `inline` 组件、<Color c="blue">强调文字</Color>，或者 <strong>加粗内容</strong>。

## 14. Callout

`Callout` 适合放短提示、边界条件和操作提醒。现在这套样式更收敛，主要靠细边框、淡背景和标题提示来区分内容。

<Callout title="补充说明">
  这类块不适合承载很长的正文，更适合放一两句和当前段落紧密相关的背景信息。
</Callout>

<Callout tone="tip" title="写作建议">
  里面依然可以写 <Color c="green">行内强调</Color>、`inline
  code`，也可以放普通链接 <a href="https://imtaotao.me" target="_blank" rel="noreferrer">imtaotao.me</a>。
</Callout>

<Callout tone="warning" title="谨慎处理">
  用在迁移、删除、覆盖这类有代价的操作前，提醒读者先确认前置条件。
</Callout>

<Callout tone="success" title="完成状态">
  适合放流程结束后的短结论，让读者快速知道这一段最后得到什么结果。
</Callout>

<Callout tone="error" title="出现错误">
  用在失败结果、不可继续的状态，或者需要读者立即修正的问题。
</Callout>

## 15. Steps

`Steps` 更适合教程、排障流程和操作说明。和普通有序列表相比，它更稳定，也更容易做统一样式。

<Steps title="发布一篇新的 demo 文章">
  <Step title="先准备文章目录">
    文章放在博客目录下，图片等本地资源尽量和文章放在一起，这样相对路径最省心，也不容易写错。
  </Step>
  <Step title="普通 markdown 不够时再上 MDX 组件">
    像 <code>ChatThread</code>、<code>Poem</code> 或媒体嵌入这种能力，优先用在能明显提升结构清晰度的地方，不要为了 demo 而 demo。
  </Step>
  <Step title="稳定后再做一次真实验证">
    等正文结构差不多定下来，再跑一次完整构建或预览。很多导入路径和 MDX 写法问题，靠肉眼看没有实际渲染来得快。
  </Step>
</Steps>

<Steps title="不用 Step 标签的写法">

- 保持 markdown 写作节奏
- 需要更细控制时再换回 `<Step>`

</Steps>

## 17. 后续约定

- 单图优先用 Markdown 原生语法
- 多图排版用 `ImageGallery`
- 外站音频入口用 `AudioEmbed`
- 外站视频入口用 `VideoEmbed`
- 段落内的音频小入口用 `AudioLink`
- 段落内的视频小入口用 `VideoLink`
- 网页 iframe 卡片用 `WebEmbed`
- 普通外链用 Markdown 链接
- GitHub 账号提及用 `GitHubMention`
- GitHub 仓库入口用 `GitHubRepo`
- 数学表达式用 `MathExpression`，块级公式加 `display="block"`
- 对话记录用 `ChatThread`
- 古诗词卡片用 `Poem`
- `columns={2}` 适合四宫格
- `columns={3}` 适合六宫格或九宫格
- `summary` 现在是可选的，不写也能发文
