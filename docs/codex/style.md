# 本地代码风格

这个文档用于记录仓库内的代码约定。

## 原则

- 优先遵循仓库现有模式。
- 改动保持最小，并聚焦当前任务。
- 可读性优先于花哨写法。
- 当逻辑风险足够高时，再补测试。
- 可复用工具优先放在 `packages/kernel-shared/`，不要散落在 `app/` 内。

## 通用约定

- 在手动调整风格之前，先尊重仓库现有格式化结果。
- 逻辑分组之间保留一个空行。
- 导入顺序：第三方、内部别名、相对路径。
- 路径更清晰时，优先使用 `tsconfig.json` 里的别名。
- 需要时使用 `pnpm format` 或 `pnpm format:md` 做格式整理。

## 命名

- 变量和函数：`camelCase`
- 类型、接口、组件：`PascalCase`
- 真正常量：`SCREAMING_SNAKE_CASE`
- 布尔值优先使用 `is`、`has`、`can`、`should` 这类前缀

## TypeScript

- 避免使用 `any`，需要收窄时优先用 `unknown`
- 这个代码库里数组类型优先写成 `Array<T>`
- 局部明显可推导的返回类型，让 TypeScript 自动推导
- 对外暴露或较复杂的 API，补足显式类型
- 如果对象映射或表驱动能让分支更清晰，优先用它们

## React

- 组件尽量保持以 props 驱动为主。
- 副作用放在 `useEffect`。
- 事件处理函数使用 `handleXxx` 命名，回调 props 使用 `onXxx`。
- 列表渲染使用稳定唯一的 key。
- 对加载、空态、错误态优先使用提前返回。

## 注释

- 注释解释“为什么”，不要解释显而易见的动作。
- TODO 要说明缺口原因和后续处理方向。

## 输出要求

- 改代码时优先做最小可行 diff。
- 避免无关的整文件格式化噪音。
