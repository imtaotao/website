最近刚看了一些 AI 相关的东西，有一些想法，比较发散，正确与否，诸君自行辨别。

## 关于概率性的输出

基于 LLM 的应用，天然有一个问题，就是输入对应的输出并不总是固定的。这本质上是一个概率问题，也就导致它在一些非常需要固定答案的业务场景里并不是那么适合，往往需要人工干预，或者有一套兜底方案。

实际上对于用户来说，很多时候 90% 的正确和 1% 的正确没有太大的区别。所以以往我一直觉得，搞 LLM 应用没什么特别好玩的，因为我需要的是固定的、确切的 output。

但是今天我突然意识到，也许我一开始的想法就是错的。LLM 的概率性输出，就像异步函数一样，是会一直污染系统的。我们想要做到理论上的 100% 正确，可能本来就是不可能的事情。但如果能持续往越来越高的正确率前进，这本身依然有非常大的应用空间。

## 提高准确率的方向

如果要提高准确率，一个很直接的方向，就是不断增加限制条件，把发散的 input 约束在一个更小的场景里，然后尽可能用更多的测试、测评 case 去反复检测，保证正确率。

也就是说，问题越收敛，变量越少，系统就越有机会表现稳定。

## 关于特征提取和标签化

至于如何保证检测结果，最开始我的想法很自然，就是从业务场景里提取特征，再把它们标签化。

例如输入一句话：“给我推荐一些好看的书籍。”

这里面可以把“推荐”“好看的”“书籍”这些信息 token 化，再让大模型基于这些 token 去进行匹配，最终找到一个相对合适的结果。这个过程可以基于 LLM 来做，也可以基于自研算法来做，本质上就是一种向量化和结构化的处理方式。

换句话说，越细化，变量因素越少，复杂度就越低，准确性也就越容易上升。我们把大的 context 片段化，这应该已经是现在社区里比较常见的手段之一，也是很多 RAG 框架里很重要的一部分。

## token 如何切分

接下来更关键的问题，其实是 token 化到底要如何切分。

这一块我暂时还没有特别清晰的想法，可能真的需要某个垂直领域里足够专精的人，才能拆得足够好。比如快递行业，如果对业务本身非常熟，就更有可能把它的业务特性拆得准确、自然、可计算。

我自己感觉，这里可能反而才是关键。

## 不必执着于 100%

只要我们的应用能够保证很高的准确性，哪怕不是 100%，依然可能为社会带来巨大的成本降低。

也许真正重要的，并不是追求一个绝对不会出错的系统，而是找到一条可以持续逼近高正确率、并且成本可控的路径。
