## 💰 价值所在

### 1. 现代大型 Web 工程开发需要什么

#### 导读

> 现在的 Web 项目，基本上复杂度是随着业务形态逐渐提升的。React/Vue 之类的框架解决了一些组件复用的问题。
> 这里组件的概念可以再扩展一些，不只是平时开发意义上的 React Component。它应该是一个集合的概念：有功能集合，就能够复用，而复用的程度取决于这个集合的干净程度。
> 但是业务形态是复杂的，所以为了应对这种复杂度，一个集合如何设计好输入输出，是很考验开发者水平的。

你要问我如何提高开发者水平，我觉得要学会懒，也不要太懒。

1. 懒，就是不要做重复无意义的事情，想办法将事务抽象起来复用。这是认识的先决条件，如果连这个想法都不存在，是不太适合理工类工作的，搞艺术文化还行。艺术是高度自由的，是有个性的美，但是需要天赋。
2. 不要太懒，是说事务如何抽象、有哪些手段、有哪些先进思想，是需要主动了解的。了解的手段有很多，你可以问同事朋友，去百度，可以 Google，去 GitHub 看高手们写的代码，去国内外社区和大佬吹牛交流等等，要勤快。聪明的人自然有高效的手段，我这方面实在是不擅长，做了一些，但不是很勤快，我读书读得不好与此也有很大的关系。

#### 职务形态：业务与基建

现在的开发者或者老板们很会固化一些词，比如基建、业务。但是基建和业务到底是什么，要做什么事情，技术之上的业务又是什么，我想单纯停留在这些词上是没有用的。

大型的 Web 工程一定伴随着复杂的业务系统，而复杂的业务系统来自用户真实的诉求。用户的诉求体现在人们日常的生活中，比如去买个东西，怎么买，怎么挑选货物等。

在 AI 出奇迹之前（GPT 这种我是不认可的），将琐碎、随机性的行为固化成需求，而且各个需求之间的联动越来越多，那么底层抽象出来的单元就会越来越多、越来越复杂，抽象层会逐渐增加。

代码的编写过程也是一个复杂的事情。如何将这些抽象层以稳定、高扩展性、体验易用的手段开发出来，这就是基建要做的事情。

多提一嘴：复杂度这么高，应对的方案也越多，上下文也越多，对新人是很不友好的。如果在高复杂度的情况下，让比较蠢的新人也能很快上手融入进去，是需要一些水平的（我还没有仔细想过）。

基建也分为两种类型：

1. 对这些能力做得越通用，则难度越高，也越属于底层类型的基建。它的收益很大，开发成本和风险也很高，但是可以对一个事情了解得更加全面，会有一种尽在掌握之中的安全感。
2. 另一种基建类型是服务于某个具体项目，在前面说的「如何设计好输入输出」的地方简化一下。比如一些静态化的参数，本来是要通过某种手段分析出来才能使用的，这种可以对所有项目生效。但是现在是某个具体业务，这些数值属性就可以固化写死，省去分析步骤，这样难度降低了，开发成本也降低了。这是一种基建与业务中间形态的基建类型。这种类型的基建做的事情不太内聚，可能会随着业务复杂度变得非常分散，好处是可以在一个团队很好地生存下去。

#### 业务型选手

1. 我一直觉得，业务型选手在于了解自己在做什么样的事情，服务于什么样的客户，要有使命感和对自身业务形态的信仰和认同，否则是无法很好地完成既定工作的。
2. 这不是说不需要关注技术本身，而是在如何让业务开发完成既定目标的过程中，沉淀出一些高度好用的东西。比如一些小的工具库、调试方法都可以。当这些沉淀足够多的时候，就会质变成一套完整的解决方案。
3. 除开业务到技术的关系之外，那就是和上层的联系了。如何去感受上层 UI/UX 设计师在设计一个产品时的思考，感受我们的产品如何影响用户，我们产品提供的功能如何高效、好用地满足用户诉求，都是需要思考的。一种是大众向的，我们提供的产品……

#### 功能点

上面说了我们的 Web 工程为什么复杂，以及作为开发者有几种工作状态。那到一个真实的前端工程里面，我们到底需要什么呢？

1. 效率优先。
2. 性能优先。
3. 扩展性优先。
4. 多平台、多运行时的支持。
5. 高效舒适的多人协作机制。

不过这几种特性其实是相辅相成的，当一种做好时，会给另外几种特性带来强有力的支撑。

## 📦 前端模块化之路

### 13. 前端模块化之路的曲折历史

参考资料：

- https://yingkaixiang.github.io/blog/JavaScript/module-history.html
- https://github.com/myshov/history-of-javascript

模块化的发展大概经历了这些阶段：

1. 一个脚本走天下，利用命名空间做隔离，模拟模块。Rollup 这些 builder 构建的产物也可以看看，其实也是通过命名空间来隔离不同模块里面的变量名。
2. 社区探索 AMD，一个函数一个模块，利用函数作用域。
3. RequireJS、Sea.js 的探索。
4. Node.js 和 CJS 的出现。
5. ESM 的出现，以及相比 CJS 的静态化优势。

ESM 这里可以展开几个点：

1. 基于现在社区对 ESM 的静态分析，可以反过来启发我们做框架语法设计、工程设计。其他编程语言也有类似情况，比如类型标注、Rust 生命周期标注。
2. 不好的地方是效率问题，以及统一的 unit 不方便包装。
3. async ESM import/export，异步的 export 很困难，尤其对打包器来说。（详细原因待写）

### 18. 可行性讨论和预期收益

业务逻辑可以抽象成模块，然后成为一个独立的 bundle。这个 bundle 不一定是 JS bundle，也可以混合其他资源。

比如在前面讨论的一些打包工具里，Webpack 就将任意资源都看待成 module，最终产物可以是一个纯粹的 JS bundle。

其实这个说法就是：业务的一个独立聚合单元最好是一个类似 npm package 的东西。至于用什么办法 import 它倒是都可以，只要性能和可用性能够达标就行。

上面这段话的意思是，ESM 现阶段还无法成为一个很好的业务型模块化单元。即使像字节内部有 PDN 这种服务，社区也有一堆方案（名字我忘记了，懒得找了），所以像前面列举的一些微前端和 Monorepo 方案，都是粒度更粗的单元，它们都能够兼具抽象和业务复杂度这两个因素。
